# 現在アプリの説明（日鋼団地 住戸希望チェック）

## 1. このアプリの目的

このアプリは、日鋼団地の住戸希望をオンラインで集めるためのWebアプリです。  
利用者は希望住戸を最大3件まで選び、団地全体の希望集中を確認できます。  
現地での長時間調整を減らし、希望の重なりを早く見える状態にすることが狙いです。  

## 2. 想定ユーザー

- **参加者**：住戸希望を提出する住民や入居予定者です。  
- **管理者**：提出状況の集計を確認する運営側です。  
- **見学者**：登録せずに操作感だけ試す人です。  

## 3. 提供している画面

- **トップ**（`/`）：利用導線の入口です。  
- **お試し入力**（`/trial`）：本番データに影響しない体験画面です。  
- **利用開始設定**（`/login`）：参加者登録と星座匿名名の選択を行う画面です。  
- **本入力**（`/units`）：希望1から希望3を選ぶ主画面です。  
- **希望可視化**（`/overview`）：希望数のヒートマップを確認する画面です。  
- **チャット**（`/units/chat`）：全体調整と部屋別調整の会話画面です。  
- **資料一覧**（`/materials`）：PDFとスクリーンショットを閲覧する画面です。  
- **サイトマップ**（`/sitemap`）：画面一覧の説明ページです。  
- **管理画面**（`/[adminSlug]`）：非公開URLで開く管理者向け画面です。  

## 4. 参加者フロー

参加者は、まず利用開始設定でユーザー名と星座匿名名を登録します。  
次に本入力画面で希望住戸を選び、提出ボタンでサーバーへ反映します。  
提出後は全体ヒートマップで混雑を見ながら、必要時にチャットで調整します。  

## 5. 本入力画面の仕様

本入力画面は、住戸選択の中心機能です。  
希望枠は1件から始まり、操作で最大3件まで追加できます。  
同一ユーザー内で同じ住戸を重複選択することはできません。  
希望カードはドラッグアンドドロップで順位を入れ替えできます。  
提出前はローカル編集状態で、提出ボタンを押した時点でサーバーへ反映されます。  
各希望には街区、棟、階、部屋のフィルタがあり、図面クリックでも同じ希望枠に反映されます。  
図面では住戸サイズ帯の色、希望件数バッジ、選択順位バッジを同時表示します。  
選択住戸の詳細として住戸番号、タイプ、面積、予定価格を表示します。  

## 6. チャット機能の仕様

チャットには全体調整チャットと希望順位別の部屋チャットがあります。  
参加者の表示名は既定で星座匿名名です。  
確認コードが有効な参加者だけ、部屋ごとに表示名を住戸/ユーザー名へ切り替えできます。  
チャット内容は現在クライアント内状態で管理され、永続保存はしていません。  

## 7. 希望可視化の仕様

ヒートマップはAPI集計結果をもとに、住戸ごとの希望件数を色で表示します。  
表示範囲は第1希望のみ、第1+第2希望、第1から第3希望の3種類を切り替えできます。  
更新時刻と集計対象セッション数を画面で確認できます。  

## 8. 管理機能の仕様

管理画面は環境変数で設定した非公開スラッグURLにだけ表示されます。  
管理者パスワード認証が通ると、専用Cookieで管理セッションを保持します。  
管理画面では提出件数、提出人数、上位人気住戸、最近の提出ログを確認できます。  

## 9. データの持ち方

- **クライアント側**：ユーザーID、ユーザー名、星座匿名名、表示設定、確認状態をLocalStorageに保存します。  
- **API側**：希望提出データはサーバープロセス内メモリのMapと配列で保持します。  
- **集計API**：提出済み希望から住戸別件数を計算して返します。  

サーバーメモリ保持のため、プロセス再起動で提出データは消えます。  

## 10. 住戸データの扱い

住戸の基本構成は、北4棟と南4棟、各14階、1フロア4ラインで定義しています。  
住戸詳細はPDF由来のレコードを使って補強しています。  
一部住戸はOCR不足として欠損検知され、画面上で手動補完対象として表示されます。  

## 11. いまの実装上の位置づけ

このアプリは実運用想定の画面体験を備えたモック段階です。  
選択UI、提出導線、集計表示、管理確認の主要フローは実装済みです。  
一方で、希望提出の永続化、チャット永続化、認証基盤の統合は未実装です。

# 統合チェックポイント ワークフロー

このワークフローは以下の2フェーズを順番に実行します。

1. **Phase 1: ボーイスカウトルール** — コードを整理・修正してクリーンな状態にする
2. **Phase 2: ブランチ同期** — クリーンなコードを `main`, `dev`, `anti` の3ブランチへ反映してGitHubに保存する

**重要**: Phase 1 が完了してビルド・テストが通っていることを確認してから Phase 2 へ進んでください。

---

# Phase 1: ボーイスカウトルール

## Role

あなたは経験豊富なシニアテックリード兼コードレビュアーです。
現在開発中のアプリケーション全体の健全性を診断し、品質を向上させるタスクを実行してください。

## Mission

以下の4つのステップで、現在のコードベースを監査・修正してください。

1. **構造・配置・登録の監査 (Structure & Registration Audit)**
2. **自動化ツールによる診断 (Automated Diagnostics)**
3. **ルール適合性チェック (Rule Compliance)**
4. **ボーイスカウトルールの適用 (Boy Scout Rule)**

## 詳細手順

### 1. 構造・配置・登録の監査

プロジェクトのファイルツリーを確認し、以下の観点で評価・修正してください。

- **配置の適切性**:
  - ファイルの責務に対して、配置されているディレクトリは適切か？ (`.agents/rules/directory-structure.md` 参照)
  - 「なんとなく」置かれているファイルや、不自然な依存関係はないか？
  - > **アクション:** 配置が明らかに間違っているファイルがあれば移動を提案、または確信があれば移動してください。

- **開発ダッシュボードへの登録**:
  - 現在チェック中のコンポーネントが「ページ (Page)」または「主要なUI機能」である場合、開発リンク集 (`/debug-links`) に登録されているか確認してください。
  - > **アクション:** 未登録の場合は、`allSections` 配列の適切なカテゴリに新しいルートオブジェクトを追加してください。
    ```typescript
    // 追加例
    {
      title: "ページタイトル",
      path: "/page-path",
      desc: "ページの説明文。",
      badge: "New" // 必要に応じて
    },
    ```

### 2. 自動化ツールによる診断

以下のコマンドを実行し、機械的に検出可能な問題を特定してください。

- **静的解析 (Linter)**:
  - > **アクション:** 検出されたエラーは必ず修正してください。警告は **Boy Scout Rule に従って段階的に減らす** 方針で扱い、今回触っているファイル・周辺ファイル・安全に直せるものを優先して修正してください。特に `no-unused-vars` や `no-console` などは積極的に修正対象とします。
  - > **重要:** 既存コードベース全体の警告を一度にゼロにすることは要求しません。今回の作業で安全に解消できる警告を 1 件でも減らすことを成功とみなします。
  - > **禁止:** 警告を黙らせるためだけの安易な `eslint-disable`、不要な `any`、意味のないコード移動で件数だけを減らす行為は禁止です。
  - > **環境注意（Windows PowerShell）**: `grep -r` は動作しません。代わりに `Select-String -Recurse -Path "src/**" -Pattern "<パターン>"` を使用してください。

- **テスト実行 (Test)**:
  - 関連するテストファイルがある場合、テストを実行してください。
  - コマンド例: `pnpm test src/components/path/to/component`
  - > **アクション:** テストが失敗している場合は、修正が可能であれば修正し、困難な場合はIssueとして報告してください。

- **一時ファイル・ログファイルの削除**:
  - 開発中に生成されたログファイル・レポートファイル（例: `*.log`, `*_report.txt`, `final_lint_report.txt`, `ai_test_output*.txt`, `ai_error.json` など）を確認してください。
  - > **アクション:** ルートに不要ファイルがある場合は **`.agents/hooks/cleanup-temp-files.md`** のコマンドを実行して削除してください。
    > 削除 / 移動先の分類基準は **`.agents/rules/directory-structure.md` § 9「ルートディレクトリ配置基準」** を参照してください。
  - > Git管理外のファイル（`.gitignore` 対象）であっても、ワーキングディレクトリを汚染している場合は削除します。

### 3. ルール適合性チェック

プロジェクトのコンテキストに含まれる【開発ルール/Guidelines】を参照し、違反しているコードがないか確認してください。

- **Accessibility (a11y)**:
  - 画像に `alt` 属性はあるか？
  - ボタンやインタラクティブ要素に `aria-label` は適切か？
  - 色のコントラストは十分か？（目視確認）
- **Security**:
  - ハードコードされたAPIキーや認証情報はないか？
  - `dangerouslySetInnerHTML` が不用意に使われていないか？
- **React Best Practices**:
  - リストレンダリングに一意な `key` 属性が使われているか？
  - `useEffect` の依存配列は正しいか？
  - 不要な再レンダリングを引き起こすようなコードはないか？
  - Next.js 16ルールに従い、`default.tsx`の設置や非同期リクエストAPIへの移行がなされているか？

### 4. ボーイスカウトルールの適用 (重要)

コードを読み込む際、**「来た時よりも少しだけ綺麗にする」**というボーイスカウトルールを適用してください。
ただし、**機能変更や大規模なリファクタリングは禁止**です。以下の範囲で安全な修正を行ってください。

- **ESLint 警告の漸進的返済**:
  - 今回の変更に直接関係するファイルに警告がある場合、可能な範囲で一緒に直してください。
  - 関連ファイルに既存警告が複数ある場合でも、全件解消は不要です。安全に直せるものを少しずつ返済してください。
  - 直せなかった警告は、件数・対象ファイル・理由を簡潔に報告してください。
- **クリーンアップ**:
  - 未使用のインポート削除
  - 不要なコメントアウトされたコードの削除
  - 誤字脱字の修正
  - Tailwind CSS v4 クラスへのマイグレーション（例: `bg-gradient-to-*` → `bg-linear-to-*`、`flex-shrink-0` → `shrink-0` 等）
- **型安全性**:
  - 不適切な `any` 型の修正（推論可能な場合は型を定義するか、必要に応じて `unknown` に置き換える）
  - `better-auth` のフック類の誤ったインポート（`client` ではなく `react` を使用する等）の修正
- **ドキュメンテーション**:
  - 可読性の低いコードへのコメント追加（日本語）
  - JSDoc/TSDoc の追加・修正
- **フォーマット**:
  - インデントやフォーマットの乱れ修正（`pnpm format` 推奨）

## Phase 1 の出力

Phase 1 の完了時に以下を報告してください。

1. **[Audit Report]**: 構造・登録状況・ルール違反に関する懸念点（修正が必要な大きな問題）のリスト。
2. **[Boy Scout Plan]**: 今回ついでに修正した「小さな改善点」のリスト。ESLint 警告を扱う場合は、どの警告を今回返済したかを明記してください。
3. **[Warning Delta]**: 修正前後で ESLint の警告がどう増減したかを簡潔に報告してください。
4. **[Go/No-Go]**: ビルドとテストが通っているかを明示し、Phase 2 へ進んで問題ないかを判定してください。

> **No-Go 判定の場合**: Phase 2 はスキップし、ユーザーに問題を報告して対処を求めてください。

# Phase 2: ブランチ同期（main / dev / anti）

## 前提

Phase 1 の **[Go/No-Go]** が **Go** であることを確認してから開始してください。

## 目的

`main`, `dev`, `anti` の3つのブランチが全く同じコミット履歴（状態）になるように、相互にマージして同期（シンクロ）させ、GitHubに保存します。
ユーザーは一人開発であり、使用するツールごとにブランチを使い分けています。

## 追加機能（必須）

- `package.json` の `version` を基準にタグ（例: `v0.4.3`）を作成できること
- 3ブランチが既に同一コミットなら、同期処理をスキップして「タグ付けのみ」実行できること
- 3ブランチのうち1つだけ不一致なら、進んでいる側（2つで一致している側、または最新コミット側）へ寄せて同期できること
- 同名タグが異なるコミットを指している場合は、`package.json` の version を **minor で必ず引き上げて** 新タグを作成すること（例: `0.14.0 -> 0.15.0`、既存タグの付け替えは禁止）

## 前提条件と注意点

- ユーザーは `git worktree` を使用している可能性があります（例: `anti` ディレクトリにいる）。
- worktree環境では、別ディレクトリでチェックアウトされているブランチに `git checkout` できない場合があります。
- そのため、このワークフローでは直接 `checkout` せず、現在のブランチで全てを統合し、その結果をリモートの各ブランチに `push` する安全な方法を取ります。
- `git branch -f` は worktree 使用中ブランチに対して失敗しうるため、失敗してもリモートSHA一致を優先して完了判定する。
- **正解コミット（canonical）は常に「現在いるブランチの `HEAD`」とする。**
- 特定ブランチ（例: `dev`）を固定で正としない。ユーザーがどのブランチで実行しても、その時点の `HEAD` を正として `main/dev/anti` を合わせる。

## 手順

### Step 1: 作業状態の確認

- 現在のディレクトリで `git status` を実行し、未コミットの変更がないか確認します。
- Phase 1 で修正が発生した場合は、ここでコミットしてください。

```bash
# Phase 1 での変更をコミット（変更がある場合）
git add -A
git commit -m "chore: ボーイスカウトルール適用 - クリーンアップ"
```

### Step 2: 最新情報の取得

```bash
git fetch origin
```

### Step 3: ブランチ同期状態の判定

```bash
git rev-parse origin/main
git rev-parse origin/dev
git rev-parse origin/anti
```

判定結果に応じて以下へ分岐：

- **3つ同一**: Step 7（タグ処理）へ進む（同期処理は不要）
- **1つだけ不一致**: Step 4-B（片側追従同期）へ進む
- **それ以外**: Step 4-A（相互マージ同期）へ進む

補足: `origin/main`, `origin/dev`, `origin/anti` が同一でも、現在の `HEAD` と異なる場合は Step 5 を実行して `HEAD` に揃える。

### Step 4-A: 現在のブランチへの統合（相互マージ）

```bash
git merge origin/main origin/dev origin/anti
```

> マージコンフリクトが発生した場合は作業を中断し、ユーザーに状況を伝えてコンフリクト解決を行ってください。

### Step 4-B: 1ブランチのみ不一致時の追従同期

原則として現在の `HEAD` を正として不一致ブランチを合わせます。

```bash
git push origin <正SHA>:main
git push origin <正SHA>:dev
git push origin <正SHA>:anti
```

### Step 5: 統合結果の各ブランチへの反映 (Push)

```bash
git push origin HEAD:main
git push origin HEAD:dev
git push origin HEAD:anti
```

> non-fast-forward で拒否された場合は、`--force-with-lease` を使って安全に同期します。

### Step 6: ローカルブランチのポインタ更新（可能な場合）

```bash
git branch -f main origin/main
git branch -f dev origin/dev
git branch -f anti origin/anti
```

> Worktree の影響で失敗する可能性があります。失敗してもリモートSHA一致を優先して完了判定します。

### Step 7: package.json の version とタグ整合

```bash
# version 確認
node -p "require('./package.json').version"

# 既存タグ確認
git tag -l "v<version>"

# タグ作成（新規の場合）
git tag -a v<version> -m "release: v<version>"
git push origin v<version>
```

既に同名タグが存在する場合:
- **同一コミット**: 成功扱いでスキップ
- **異なるコミット**: version を **minor で引き上げる（必須）**
  1. `pnpm version minor --no-git-tag-version`
  2. `package.json` 変更をコミット
  3. 新しい version で `v<newVersion>` タグを作成して push
  4. 既存タグの削除・付け替えは行わない

補足:
- このチェックポイントでは、タグ競合時の `patch` バンプ（例: `0.14.0 -> 0.14.1`）は禁止。
- ユーザーが明示的に別ルール（major/patch）を指定した場合のみ、その指定を優先してよい。

## Phase 2 の出力

完了時に以下を必ず報告してください。

1. `origin/main`, `origin/dev`, `origin/anti` の最終SHA
2. `package.json` の version
3. 作成/利用したタグ（例: `v0.4.3`）
4. `git branch -f` の失敗有無（失敗時はworktree制約として記録）
5. **ワークフロー完了サマリー**: Phase 1 と Phase 2 の実行結果を一覧で表示

# 制約事項

- 既存の機能（挙動）はいかなる場合も変更・破壊しないこと。
- Phase 1 では機能変更や大規模リファクタリングは禁止。
- 迷った場合は勝手に修正せず、ユーザーに確認すること。
- Phase 2 は Phase 1 の Go 判定後にのみ実行すること。

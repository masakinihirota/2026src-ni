# Web Security Check Workflow

このワークフローは、Webアプリのセキュリティを実務観点で厳格に監査します。
`review-security-core` の基盤監査に加えて、以下の知見を明示的に取り込みます。

- `sample/_アイデアメモ/_セキュリティチェックポイント.md`
- `sample/_アイデアメモ/_徳丸本まとめ.md`

攻撃者視点（Adversarial Review）で、認証・認可・入力検証・出力エンコーディング・運用まで一貫して確認します。

## 対象ディレクトリ・ファイル（推奨）

- `src/lib/`（認証、認可、ユーティリティ、DBアクセス）
- `src/app/api/`（Route Handlers）
- `src/proxy.ts`（境界防御）
- `src/components/**/*.logic.ts`（データ取得・副作用層）
- `drizzle/`（RLS、マイグレーション、制約）

## レビュー前提

1. `code-review` スキルを適用する。
2. 次の2文書をレビュー観点として必ず参照する。
   - `sample/_アイデアメモ/_セキュリティチェックポイント.md`
   - `sample/_アイデアメモ/_徳丸本まとめ.md`
3. 「ログインできる」と「他人のデータを見られない」を分離して検証する。

## 手順

1. **対象の特定**
   - 変更差分を起点に、認証/認可/入力境界/データ境界に到達する依存を展開する。

2. **必須チェック（Webセキュリティ）**
   - **認証・セッション**:
     - セッション固定化対策（認証後のセッション再生成）があるか。
     - Cookie属性（Secure/HttpOnly/SameSite）が適切か。
   - **認可・IDOR**:
     - サーバー側で毎リクエスト所有者検証をしているか。
     - URLパラメータ改ざんで他人データへアクセスできないか。
   - **入力検証**:
     - クライアント側だけでなく、Route Handler/Server Actionでスキーマ検証しているか。
     - ファイルアップロード時に拡張子・MIME・マジックナンバー・サイズ・保存先を検証しているか。
     - パストラバーサル（`../` など）を防ぐ保存パス正規化/検証があるか。
   - **インジェクション対策**:
     - SQLインジェクション（プレースホルダ/ORM安全利用）。
     - XSS（HTML/属性/URLコンテキスト別エスケープ）。
     - コマンドインジェクション/ヘッダインジェクション/オープンリダイレクト。
   - **CSRF/CORS/クリックジャッキング**:
     - 重要操作のCSRF対策。
     - CORSの過剰許可（`*` + credentials）有無。
     - `X-Frame-Options` または `frame-ancestors` の設定。
   - **セキュリティヘッダ**:
     - `Strict-Transport-Security`（HSTS）が設定されているか。
     - `X-Content-Type-Options: nosniff` が設定されているか。
     - `Content-Security-Policy` が適切に設定されているか。
     - `Referrer-Policy` が設定されているか。
   - **エラー・ログ**:
     - 本番でスタックトレースや内部情報を露出していないか。
     - 認可失敗や異常アクセスが監査可能な構造化ログで残るか。
     - ログに機密情報（鍵、トークン、PII）を出力していないか。
     - ログ保存先がアプリ実行環境と分離され、改ざん耐性があるか。
     - 不正アクセス兆候を検知するアラートが設定されているか。
     - ログレビュー（週次/月次など）の運用手順が定義されているか。
   - **依存関係・運用**:
     - 依存ライブラリの実在性、保守性、脆弱性対応状況。
     - 環境変数/Secrets管理がハードコードなしで運用されているか。

3. **基盤整合チェック（core相当）**
   - RSC/API境界、Next.js 16の非同期リクエストAPI適用、RLS有効化、N+1/過剰取得を確認する。

4. **レポート作成**
   - 重大度を `[Critical] / [Major] / [Minor]` で分類し、以下を必須記載する。
     - 問題の要約
     - 影響範囲
     - 再現手順
     - 根本原因
     - 修正案（必要に応じて Zod スキーマ例を含む）
     - 再発防止策（テスト、ルール、監査）

## 出力フォーマット（必須）

1. Findings（重大度順）
2. Open Questions / Assumptions
3. 追加の推奨テスト
4. 変更提案サマリー

## コマンド例

- `/review-security-web-check`
- `/review-security-web-check src/app/api`
- `/review-security-web-check src/lib`

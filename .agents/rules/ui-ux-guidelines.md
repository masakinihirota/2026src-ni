# UI/UX & Design Guidelines

ユーザーインターフェースのデザイン、テーマ、およびユーザーフィードバックに関するガイドラインです。

## 共通テンプレート

### 目的
- UI表現・体験品質・アクセシビリティ基準を統一し、画面品質のばらつきを抑える。

### 必須
- 単色ベースの情報設計と十分なコントラストを確保する。
- 平易で丁寧なエラーメッセージを提供する。

### 禁止
- 装飾目的のグラデーション使用。
- 威圧的または専門用語中心のユーザー向け表現。

### 例外
- データ可視化や明示承認済みブランド素材でのみ例外を許可する。

## 1. Design Theme (デザインテーマ)

デジタル庁UIの構造思想を取り入れつつ、視覚的には以下のテーマを適用します。

### Light Mode: Glassmorphism (グラスモーフィズム)

- **コンセプト**: 明るく柔らかい配色と、ガラスのような半透明・ぼかし効果。
- **背景**: 明るい単色または近似トーン（例: `bg-slate-50`, `bg-white`）。
- **カード/モーダル**: `bg-white/10 backdrop-blur-md border border-white/20 shadow-lg`。
- **アクセント**: ネイビー/ゴールド系（例: `#1E3A8A` / `#B45309`）。
- **制約**: 可読性を最優先し、コントラスト比4.5:1以上を確保すること。

### Dark Mode: Elegant (エレガント)

- **コンセプト**: 上品で控えめな暗色、タイポグラフィ重視。
- **背景**: 深いニュートラル（例: `bg-[#0B0F1A]`）。
- **カード**: 上品な暗色ガラス（例: `bg-white/5 backdrop-blur-md`）。
- **テキスト**: `text-neutral-200` を基本とし、白(#FFF)の使用は避ける。
- **実装**: `.dark` クラスとCSS変数（トークン）を用いて実装・切替を行う。

## 2. Implementation Strategy (実装戦略)

### グラデーション利用ルール（必須）

- **原則禁止**: 装飾目的のグラデーション（背景、ボタン、見出し文字、枠線、アイコン）を禁止する。
- **禁止クラス例**: `bg-gradient-*`, `from-<color>`, `via-<color>`, `to-<color>`, `text-transparent bg-clip-text`。
- **許可例外**: データ可視化（チャートの意味表現）やブランド資産として明示承認された素材のみ。
- **代替手段**: 単色 + コントラスト + シャドウ/ボーダー/タイポグラフィで情報階層を作る。

- **CSS変数**: テーマ用トークン（`--bg`, `--card`, `--text`, `--accent`）を定義し、ライト/ダークで値を切り替える。
- **Tailwind**: ユーティリティクラスを活用するが、色はトークン経由（例: `bg-[var(--bg)]` や `bg-background`）で指定する。
- **コンポーネント優先**: `Card`, `Button` などの共通コンポーネントから置換を進める。

## 3. UX Feedback & Error Messages (フィードバック設計)

高齢者やITに不慣れなユーザーに配慮し、優しくわかりやすいフィードバックを提供します。

### メッセージ記述ルール

- **専門用語禁止**: "Invalid Token", "500 Error" などは禁止。「ログイン有効期限が切れました」「サーバー側で一時的な問題が発生しました」のように平易な言葉に変換する。
- **解決策の提示**: 「失敗しました」だけでなく、「どうすればよいか」をセットで提示する。
- **トーン**: 丁寧語（「～です」「～ます」）で統一し、威圧感のある記号や色は避ける。

### 視覚的フィードバック

- **Loading**: 通信・処理中は必ずローディング（スピナーやスケルトン）を表示する。
- **トースト**: 操作完了時は必ず成功/失敗のフィードバックを表示する。
  - **成功**: 数秒で消える緑/青のトースト。
  - **失敗**: ユーザーが閉じるまで消えない赤/オレンジのメッセージ（読み飛ばし防止）。

### 実装パターン

- **フォームバリデーション**: `onBlur` や `onChange` でリアルタイムに、平易な日本語でエラーを表示する。
- **エラーハンドリング**: `try-catch` で捕捉したエラーを、ユーザーフレンドリーなメッセージに変換するユーティリティを使用する。

## 4. Selection Button Consistency (選択系ボタン統一)

トグル・フィルター・タブのような「複数候補から選択するボタン」は、画面差分で見た目がブレないように統一する。

### 必須ルール

- 同一機能群の選択ボタンは、表示ページと編集ページで同じ視覚トーンを使う。
- Active（選択中）は「薄い塗り + 境界線 + 文字強調」を基本にする。
- Inactive（未選択）は「白背景 + 境界線 + hoverで軽い変化」を基本にする。
- `cursor-pointer` を必須とする。
- 押下可能要素は、hover時に必ずポインターカーソルへ変化しなければならない。
- サイト全体の既定値は `src/app/globals.css` で担保し、個別コンポーネントでは不足する要素に `cursor-pointer` を明示する。
- 可能な限り同一ファイル内で共通クラス定数化、または `src/components/ui` の共通コンポーネント化を行う。
- **カテゴリごとの色は、[色・配色システム](../../vns-masakinihirota-design/0012%20全体%20-01要件定義書%20-02設計書%20-03テスト計画書/色%20基調色.md) Section 2.3 (Tertiary Button) に従う。**

### 禁止ルール

- 同じ役割のボタンに対して、ある画面だけベタ塗り（濃色塗りつぶし）を採用する。
- Active/Inactive のコントラスト設計が画面ごとに異なる。
- 同じ機能でクラス定義を重複記述し、将来の変更点を分散させる。
- カテゴリ色マッピング (Rose/Blue/Emerald/Orange) を無視してランダムな色を選ぶ。
- 押下可能要素なのに `cursor: default` のまま放置する。
- disabled ではない押下可能要素に、hover時の押下可能サインが存在しない。

# General Rules

## 共通テンプレート

### 目的
- 作業フロー、自己改善、検証手順を統一し、実装品質の再現性を高める。

### 必須
- 設計判断が必要な作業はプランモードで開始する。
- 完了前に検証を実施し、根拠を持って完了判定する。

### 禁止
- 根拠のない見切り完了。
- ルール外のタスク管理配置（`schedule_todo_list/` 以外）。

### 例外
- 単純かつ影響が限定的な修正では、過剰な設計手順を省略できる。

### 参照
- [厳格なコード審査基準](./strict-review-standards.md)
- [TODO管理ルール](./todo-list-management.md)

## ワークフロー設計

### 1. 計画フェーズ（プランモード）
- 3ステップ以上 または 設計判断が必要なタスクは **必ずプランモードで開始する**
- 途中で問題が発生したら **即座に作業を止めて再計画する**（無理に進めない）
- 構築だけでなく、検証ステップもプランモードで計画する
- 曖昧さを減らすため、実装前に詳細な仕様を書く
- **Hooks の確認**: ページ作成やタスク完了の条件を満たした場合、必ず `.agents/hooks/` 内の指示を確認し、実行する。
- React Hook（`useState` / `useEffect` / `useMemo` など）を含む `*.view.tsx` / `*.tsx` を追加・更新した場合は、`.agents/hooks/client-component-boundary-check.md` を必ず実行する。
- ボタン・リンク・タブ・トグルなど押下可能要素を追加・更新した場合は、`.agents/hooks/interactive-cursor-check.md` を必ず実行する。
- タブ名の追加・編集・リネーム処理を追加・更新した場合は、`.agents/hooks/tab-name-normalization-check.md` を必ず実行する。
- Hono API（`src/lib/api/routes/**` / `src/lib/api/middleware/**` / `src/app/api/[[...route]]/route.ts`）を追加・更新した場合は、`.agents/hooks/hono-api-implementation-check.md` を必ず実行する。

### 1.0 作業開始時の定型チェック

作業開始時は、以下をテンプレートとして毎回先に確認する。

1. 今回のタスク責務に対応する `.agents/rules/*.md` を確認する。
2. `src/components/**` 配下の `*.tsx` / `*.view.tsx` で React Hook を追加・更新する可能性があるかを先に判定する。
3. Hook を触る可能性がある場合は、実装前に `.agents/hooks/client-component-boundary-check.md` を開き、完了までチェック対象として固定する。
4. Hook を含む UI を barrel (`index.ts`) 経由で export している場合は、Client/Server 境界確認を実装計画に含める。
5. 実装後の最小検証に `pnpm build` を必ず含める。
6. 押下可能要素を触る可能性がある場合は、`.agents/hooks/interactive-cursor-check.md` を実装前に確認する。
7. タブ名を扱うロジックを触る可能性がある場合は、`.agents/hooks/tab-name-normalization-check.md` を実装前に確認する。
8. Hono API実装を触る可能性がある場合は、`.agents/hooks/hono-api-implementation-check.md` を実装前に確認する。

### 1.1 `// turbo` コメントの共通定義
- ワークフロー内の `// turbo` は「並列実行してよいステップ」を示す共通記法とする。
- `// turbo` が付いた処理は、依存関係がない範囲でサブエージェント委譲または並列コマンド実行を許可する。
- 依存関係がある処理（前工程の成果物を参照する処理）では `// turbo` を付けない。

### 2. サブエージェント戦略
- メインのコンテキストウィンドウをクリーンに保つために **サブエージェントを積極活用する**
- リサーチ・コード探索・並列分析はサブエージェントに委譲する
- 複雑な問題には複数のサブエージェントを投入してより多くの計算リソースをかける
- 1サブエージェント = 1タスクで集中して実行させる

### 3. 自己改善ループ
- ユーザーから修正指摘を受けたら **必ず `.agents/rules/lessons.md` にそのパターンを記録する**
- 同じミスを繰り返さないためのルールを自分向けに書く
- ミス率が下がるまで、ルールを徹底的に改善し続ける
- セッション開始時にそのプロジェクトに関連するレッスンをレビューする

### 4. 完了前の検証（必須）
- 動作を証明できるまで、タスクを完了とマークしない
- 必要に応じてmainブランチと自分の変更の差分を確認する
- 「スタッフエンジニアがこれを承認するか？」と自問する
- テストを実行し、ログを確認し、正しさを実証する

### 4.1 ルール文書と実装の差分監査（定期）
- ルール文書と実装コードの整合性差分監査を **月次で1回** 実施する。
- 監査対象の最低範囲は `.agents/rules/`、`.agents/workflows/`、`src/` とする。
- 監査で判明した乖離は、ルール更新または実装修正のどちらで解消するかを明記して処理する。

### 5. エレガントさを追求する（バランス重視）
- 重要な変更をする前に「もっとエレガントな方法はないか？」と一度立ち止まる
- 修正がハック的に感じたら：「今知っている全てを活かして、エレガントな解決策を実装する」
- 単純で明白な修正にはこれを省略する（過剰設計しない）
- 提示する前に自分のコードを批判的にレビューする

### 6. バグ修正の自律対応
- バグレポートが来たら **そのまま修正する**（手取り足取り聞かない）
- ログ・エラー・失敗テストを手がかりにして自力で解決する
- ユーザーのコンテキストスイッチをゼロにする
- 指示されなくても CI の失敗テストを修正しに行く

## タスク管理・TODOリスト管理の厳格ルール

| ステップ | 内容 |
|----------|------|
| **配置場所(絶対ルール)** | **TODOリストやタスク管理ファイルはすべて `schedule_todo_list/` フォルダ内にのみ配置する。** `tasks/` フォルダ等を作成することは絶対に禁止。 |
| **計画を立てる** | `schedule_todo_list/TODO.md` (名称は日付等に準ずる) にチェック可能な項目でプランを書く |
| **計画を確認する** | 実装を開始する前に確認する|
| **進捗を追跡する** | 完了した項目を随時マークしていく |
| **変更を説明する** | 各ステップでハイレベルなサマリーを提示し、適用した Hooks を明記する |
| **結果を記録する** | `schedule_todo_list/TODO.md` 等にレビューセクションを追加し、Hooks の実行結果も含む |
| **レッスンを残す** | 修正指摘を受けたら対象の運用ドキュメントを更新する |

## 基本原則
**シンプルさを最優先**: あらゆる変更を可能な限りシンプルにする。影響するコードを最小限に留める。
**手を抜かない**: 根本原因を突き止める。一時しのぎの修正はしない。シニアエンジニアの水準で臨む。
**影響を最小化する**: 変更は必要な箇所だけに留める。バグを新たに持ち込まない。

## コード品質基準

**厳格なコード審査基準** (.agents/rules/strict-review-standards.md) に従うこと。特にクリティカルな項目:

- 🔴 **環境変数検証**: OAuth credentials と本番環境設定は起動時に検証
- 🔴 **Database Security**: RLS ポリシー、インデックス戦略、N+1 防止
- 🔴 **Error Handling**: エラーメッセージ詳細化、セキュリティイベント記録
- 🔴 **Testing**: クリティカル機能にはテストを必須化

詳細は [Strict Review Standards](./strict-review-standards.md) を参照。

## ツール連携戦略（CLI優先）

### 基本方針

- 本プロジェクトでは、実装タスクの自動化・検証は **CLI 優先** で行う。
- MCP サーバーの新規導入・依存追加は原則禁止とする。
- 型安全な入出力が必須な場合のみ、Next.js 内部 API（Route Handler）を選択肢とする。

### 意思決定マトリックス

| 状況 | 推奨手段 |
|------|---------|
| CLI がある | CLI 呼び出し |
| 複数バリエーション必要 | `pnpm` scripts で分岐 |
| 型安全が必須 | 内部 API エンドポイント |
| Web API only SaaS | 個別検討（まず REST API を優先） |

### 禁止事項

- MCP サーバーの新規追加
- MCP 依存の導入
- MCP 認証フローの実装
- MCP との複雑な仲介層の実装

### 実装メモ

- 実行コマンドは `package.json` の scripts を優先して利用する。
- 文字列検索は `rg` 優先とするが、Windows で `rg` 未導入の場合は `Select-String` を標準フォールバックとして即時切替する（`.agents/skills/windows-powershell-rules/SKILL.md` を参照）。
- 追加ルールが必要な場合は、`general-rules.md` へ追記し `copilot-instructions.md` には要約のみ残す。

# Large File Refactoring Rules

## 分割判断の第一原則（Why to Split）

> **「何行を超えたら分割」ではなく「なぜ分割するのか」を先に問う。**

分割を正当化する理由がない限り、行数だけを根拠に分割してはならない。
以下のいずれかが成立する場合にのみ分割を開始する。

| 判断軸 | 具体例 |
|--------|--------|
| **責務の混在** | 表示・状態管理・ビジネスロジックが同ファイルに混在している |
| **変更容易性の低下** | 1か所の修正が無関係な表示ロジックに波及する |
| **テスト困難性** | 副作用と純粋ロジックが分離できずユニットテストを書けない |
| **再利用妨害** | 単一責務の関数・型が巨大ファイルに埋もれて再利用できない |

上記のいずれにも当てはまらず、読み書きしやすければ 500 行超でも分割を強制しない。

## 共通テンプレート

### 目的
- `src/components` 配下の長大ファイルを、無理な分割ではなく自然な責務境界で継続改善する。

### 必須
- 500 行は目安として扱い、まず責務境界の明確化を優先する。
- `src/components` 配下で 500 行を超えるファイルは、可能な範囲で責務ごとに分割する。
- 分割先は原則として同一機能ディレクトリ配下に置く。
- 分割後の公開面は `index.ts` などの公開 API 経由に統一する。
- 自然な責務分割が成立しない場合は、分割を強行せず理由を記録し、レビュー時に相談する。

### 禁止
- 行数削減のみを目的とした別ツリー退避（例: `src/lib/components-refactor`）での見かけ上の分割。
- 責務を崩してまで 500 行未満へ押し込むこと。

### 例外
- `src/components/ui`（Shadcn/UI 等の共通 UI ライブラリ層）は本ルールの行数上限対象外とする。
- モックデータ専用ファイル（`*mock*data*.ts` など、機能ロジックを含まないデータ定義のみのファイル）は、可読性を優先して 500 行超を許容する。
- DBスキーマ定義ファイル（`schema.postgres.ts`、`schema.zod.ts` 等）は、テーブル定義・リレーション・型がひとつの正本として集約される性質を持つため、行数上限対象外とする。
- 自動生成ファイル・外部由来で保守不能なファイルは、例外理由を PR/コミット説明に明記した場合のみ一時許容する。
- **500 行弱（目安: 500〜540 行程度）で責務が 1 つに収まっているファイルは、合理的な分割理由がない限り分割しない。** 分割するかどうかは行数ではなく「責務の混在」と「変更容易性の低下」を判断軸とする。

## 適用範囲

- 対象: `src/components/**/*.ts`, `src/components/**/*.tsx`
- 除外: `src/components/ui/**`

## 分割基準（責務単位）

以下の責務を 1 ファイルに混在させない。

1. 表示（presenter / view）
2. 状態管理（container / hook）
3. ビジネスロジック（logic / utils）
4. データアクセス（data-access / fetch / mapper）
5. 定数・型（constants / types）

## 推奨ファイル構成

```text
feature-name/
  feature-name.tsx                  # UI（表示）
  feature-name.container.tsx        # 状態管理
  feature-name.hook.ts              # Hookロジック
  feature-name.logic.ts             # 純粋関数/ドメインロジック
  feature-name.data-access.ts       # API/DBアクセス
  feature-name.constants.ts         # 定数
  feature-name.types.ts             # 型
  index.ts                          # 公開API
```

## 運用ルール

1. 新規実装時
- 200 行を超える時点で分割設計を開始する。
- 400 行を超える前に責務分割を完了する。

2. 既存修正時
- 修正対象が 500 行超なら、機能追加より先に最小分割を実施する。
- 少なくとも `logic` と `presenter` の分離を先に行う。
- 分割後に可読性や責務の自然さが悪化する場合は、分割を打ち切って相談する。

3. 完了判定
- 責務境界が明確で、変更容易性が向上している。
- 相対 import 破綻がなく、`pnpm build` が通る。
- deep import を増やしていない。
- 500 行超が残る場合は、残置理由と今後の分割候補を記録して合意する。

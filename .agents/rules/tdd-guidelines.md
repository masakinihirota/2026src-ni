# TDD Guidelines

## 共通テンプレート

### 目的
- RGRサイクルを標準化し、実装品質と回帰耐性を高める。

### 必須
- RED→GREEN→REFACTOR の順序を守る。
- テストを対象実装の近傍に配置する。
- UIではアクセシビリティ検証を含める。

### 禁止
- テストなし実装。
- GREEN未達のまま完了扱い。

### 例外
- 例外運用が必要な場合は、理由と代替検証手段を明記する。

### 参照
- [アクセシビリティガイドライン](./accessibility-guidelines.md)
- [コーディング規約](./coding-standards.md)

## 1. TDD実行手順（AIへの依頼）

必ずt-wada方式のテスト駆動開発で開発する

### 1.0 スキル適用順序（必須）

TDDを実行する場合は、以下の順序を必須とする。

1. `test-case-outline-generation`
2. `test-code-generation`

意図:

- 先に仕様アウトライン（`describe`/`it.todo`）を固定する
- 次に同じ構造を維持したまま実行可能テストへ昇格する

禁止:

- アウトライン未作成のまま `test-code-generation` を先行実行する
- 生成後に `describe` 階層やテストケース名を恣意的に改変する

AIは、選択した**細分化タスク**を完了させるため、以下のRGRサイクルを順に実行すること。

### 1.1. テストファイル命名規則

| ファイル種別       | 命名規則                                         | 用途                                 |
| ------------------ | ------------------------------------------------ | ------------------------------------ |
| **View関連**       | `*.test.tsx`                                     | UIコンポーネントのテスト             |
| **ロジック/Fetch** | `*.logic.test.ts`, `*.logic.integration.test.ts` | ビジネスロジック、データ取得のテスト |

- テストファイルは、テストするコンポーネントやロジックと同じディレクトリに配置し、関連性を明確にします。

### 1.2. テスト記述規約

| 項目                     | ルール                                                                                               |
| ------------------------ | ---------------------------------------------------------------------------------------------------- |
| **テストフレームワーク** | Vitest + React Testing Library                                                                       |
| **テスト構造**           | `describe` でコンポーネント/関数名、`it` で具体的な振る舞い                                          |
| **言語**                 | テストケース名は日本語で記述し、仕様を明確にする                                                     |
| **アクセシビリティ**     | **重要**: UIコンポーネントのテストには `vitest-axe` によるチェック(`toHaveNoViolations`)を含めること |
| **パターン**             | AAA パターン（Arrange（準備）、Act（実行）、Assert（検証））を明確に分ける                           |

**例:**

```typescript
describe("ProfileList", () => {
  it("名前を表示する", () => {
    // Arrange
    const props = { name: "山田太郎", role: "エンジニア", bio: "開発者" };

    // Act
    render(<ProfileList {...props} />);

    // Assert
    expect(screen.getByText("山田太郎")).toBeInTheDocument();
  });
});
```

### 1.3. RGRサイクル

#### 🔴 ステップ1: テストコードの生成（REDフェーズ）

1. タスクの要件を満たすための**失敗するテストコード**を、該当する**テストファイル**（例: `[ComponentName].test.tsx`）に記述する。
2. テストを実行し、**失敗（RED）すること**を確認する。

#### 🟢 ステップ2: アプリケーションコードの生成（GREENフェーズ）

1. ステップ1で生成したテストを**すべて合格させる**ための**最小限のアプリケーションコード**を、対応するファイルに記述する。
2. テストを再度実行し、**すべてがグリーン（合格）**になることを確認する。

#### ✨ ステップ3: リファクタリング（REFACTORフェーズ）

1. テストがグリーンであることを前提に、生成したコードに対して**品質、可読性、効率**を高めるためのリファクタリングを提案・実行する。
2. リファクタリング後、**テストが引き続きグリーン**であることを最終確認し、**タスクを完了（チェック）**とする。

## 2. 実行コマンド（PowerShell / Windows）

AIが自動で実行する、またはユーザーに提案するテスト関連のコマンド例です:

```pwsh
# 全テスト実行（デフォルト）
pnpm test

# テスト実行結果を JSON 形式で出力
pnpm test --reporter=json

# UI モード（ウェブブラウザでインタラクティブに検証）
pnpm test:ui

# カバレッジ確認（CI/ローカル）
pnpm test:coverage
```

**注**: スクリプト `test:run` が定義されている場合は `pnpm test:run` でも構いませんが、**`pnpm test` がデフォルト**です。

- カバレッジの閾値目安: 機能単位で 80% 以上、クリティカル分岐は 100% を目指す。
- 品質ゲート: `pnpm build`, Biome/TSの `pnpm lint`, `pnpm test` がすべてPASSすることを自己判定の基準とする。

## 3. Strict Mode テストケース設計（React特有）

React Strict Mode の二重呼び出しは、setState updater の純粋性違反や Ref の副作用を検出する強力なメカニズムです。テスト駆動開発時に、以下のパターンをカバーしてください。

### 3.1 繰り返し操作テストケース

**目的**: 同じ操作を連続実行したとき、各回で独立した状態遷移が起こることを検証

**パターン例**:
- ボタンを2回連続クリック → 2つの独立した状態変化が起こる（上書きされない）
- フォーム送信を2回 → 2回分の処理が独立して実行される
- 候補追加を複数回 → 各回が異なるスロットに追加される（同じスロットへの上書きなし）

**具体例** (このセッションで実装):
```typescript
it("+候補を追加を2回押すと候補1・候補2に別々に追加されること", async () => {
    const user = userEvent.setup();
    render(<ProfileCreationWizard />);

    // 1回目
    await user.click(screen.getByRole("button", { name: "+ 候補を追加" }));
    await waitFor(() => {
        expect(screen.getByText(/星座匿名候補1:.+/)).toBeInTheDocument();
    });

    // 2回目（候補1は上書きされない）
    await user.click(screen.getByRole("button", { name: "+ 候補を追加" }));
    await waitFor(() => {
        expect(screen.getByText(/星座匿名候補2:.+/)).toBeInTheDocument();
        expect(screen.getByText(/星座匿名候補1:.+/)).toBeInTheDocument(); // ← 重要
    });
});
```

### 3.2 テスト実行時の Strict Mode 確認

- **開発環境では必ず Strict Mode を有効**にしてテストを実行する (`next.config.ts` の `reactStrictMode: true`)
- CI/CD パイプラインでも Strict Mode 有効でテストを実行する
- Strict Mode で二重呼び出しが発生すると、不具合が顕著に現れるため、**開発時の early warning** として活用する

### 3.3 検出対象（よくある違反パターン）

| パターン | テストケース例 | 検出方法 |
|---------|---------------|--------|
| **Ref の読み書き in updater** | setState 内で ref を読み取り・クリア | 2回目呼び出しで ref の値が異なる |
| **外部変数の状態依存** | グローバル変数アクセス | 1回目と2回目で異なる計算結果 |
| **Effect 依存配列への動的値混入** | 毎回異なる参照が含まれる | useEffect が毎フレーム再実行される |
| **Event Handler の非純粋性** | onClick で Date.now() など | 同じ操作で毎回異なる値が記録される |

### 3.4 PR レビューチェックリスト

- [ ] 連続実行テストケースが存在するか確認
- [ ] Strict Mode で二重呼び出しを意図的にテストしているか確認
- [ ] setState updater が純粋関数（副作用なし）か確認
- [ ] 新機能追加時に、このテストパターンを追加したか確認

## 4. テストコード生成プロトコル（アウトラインから実装）

`describe + it.todo` のアウトラインを実行可能テストへ変換する際は、以下を必須とする。

### 4.1 構造維持

- `describe` の階層構造を維持する
- テストケース名（仕様意図）を維持する
- `it.todo` を同名の `it` へ置換して実装する

### 4.2 データライフサイクル管理（Hook必須）

- テストデータ作成は関連 `describe` の `beforeEach` で行う
- クリーンアップは同じ `beforeEach` 内で `context.onTestFinished` に登録する
- これにより失敗時も含めた Test Isolation を担保する

実装パターン:

```ts
beforeEach(async (context) => {
  const id = await createFixture()

  context.onTestFinished(async () => {
    await cleanupFixture(id)
  })
})
```

### 4.3 AAA 運用

- Arrange: `describe` スコープ / `beforeEach` で準備
- Act: `it` 内で対象動作を1回実行
- Assert: `it` 内で事後条件と不変条件を検証

### 4.4 モック方針

- 内部モジュールへの `vi.mock` / `vi.spyOn` は禁止
- DB検証は実DB + ORM を利用
- 外部HTTPは Docker 上の prism サーバを利用

### 4.5 契約検証

- 事後条件: 出力値 + DB更新後状態の双方を検証
- 不変条件: 異常系でエラー + DB非変更を検証

### 4.6 失敗時分岐

- テスト実装ミス: テストを修正して再実行
- 実装/仕様不整合: `it.skip` へ変更し、理由と確認事項をユーザーへ提示

### 4.7 実行コマンド

- scripts 経由で実行する（`pnpm test:run -- <test-file>` 推奨）
- 依頼文に `npm run test` があっても、このリポジトリでは `pnpm` を優先する

## 5. TDD開始チェックリスト（必須）

TDD着手時は、以下をすべて満たしてから実装を開始する。

- [ ] `test-case-outline-generation` を先に適用した
- [ ] `describe` 階層と `it.todo` ケース名が仕様として確定している
- [ ] 不明仕様（同率、期間境界、0件時表示など）を解消した
- [ ] `test-code-generation` で実装化する対象ファイルを特定した
- [ ] 実行コマンドを scripts 経由（pnpm）に固定した

## 6. 標準プロンプト（推奨）

TDD依頼時は、以下の定型文を推奨する。

```text
TDDとして実施してください。
必ず test-case-outline-generation → test-code-generation の順で進めてください。
まず describe/it.todo でアウトラインを確定し、次に同じ構造・同じケース名のまま it 実装へ変換してください。
失敗時は、テスト実装誤りなら修正、実装/仕様不整合なら it.skip + 理由 + 確認事項を提示してください。
```

## 7. 失敗時分岐テンプレート（固定）

### 7.1 テスト実装誤りの場合

- 現象: テストコードの前提/アサートが誤っている
- 対応: テストコードを修正して再実行
- 報告: 「修正内容」「再実行結果」「残リスク」を記載

### 7.2 実装/仕様不整合の場合

- 現象: 現行仕様では成立しない期待値、または仕様未確定
- 対応: 該当ケースを `it.skip` に変更
- 必須記載:
  - スキップ理由
  - 期待仕様（案）
  - ユーザー確認事項

`it.skip` 記述例:

```ts
it.skip("同率時に作成日昇順で並ぶこと", () => {
  // TODO: 現行実装/仕様では同率順位ルールが未確定。
  // 仕様確定後に有効化する。
});
```

# Large File Refactoring

`src/components` 配下の長大ファイルを、見かけ上の移動ではなく責務分割で改善するための運用スキルです。500 行は目安であり、無理な分割は行いません。

## 0. 分割判断の第一原則（Why to Split）

> **「何行を超えたら分割」ではなく「なぜ分割するのか」を先に問う。**

分割を正当化する理由がなければ、行数だけを根拠に分割してはならない。
以下のいずれかが成立する場合にのみ分割を開始する。

| 判断軸 | 具体例 |
|--------|--------|
| **責務の混在** | 表示・状態管理・ビジネスロジックが同ファイルに混在している |
| **変更容易性の低下** | 1か所の修正が無関係な表示ロジックに波及する |
| **テスト困難性** | 副作用と純粋ロジックが分離できずユニットテストを書けない |
| **再利用妨害** | 単一責務の関数・型が巨大ファイルに埋もれて再利用できない |

上記のいずれにも当てはまらず、読み書きしやすければ 500 行超でも分割を強制しない。

## 1. 適用シーン

- `src/components` に 500 行超ファイルがある
- 機能追加のたびに 1 ファイルへ実装が集中している
- 状態管理・表示・データアクセスが混在している

## 2. 適用範囲

- 対象: `src/components/**/*.ts`, `src/components/**/*.tsx`
- 除外: `src/components/ui/**`
- 例外: モックデータ専用ファイル（`*mock*data*.ts` 等）は 500 行超を許容
- **500 行弱（目安: 500〜540 行程度）で責務が 1 つに収まっているファイルは、合理的な分割理由がない限り分割しない。** 行数よりも「責務の混在」と「変更容易性の低下」を優先判断軸とする。

## 3. 分割ポリシー

1. 同一機能ディレクトリ内で分割する
2. 公開 API は `index.ts` に集約する
3. 以下の責務を分離する
   - 表示（presenter）
   - 状態管理（container / hook）
   - ビジネスロジック（logic / utils）
   - データアクセス（data-access / fetch）
   - 定数・型（constants / types）

## 4. 禁止事項

- 行数回避のために別ツリーへ実体退避すること
- 500 行超を「後で対応」として放置すること
- deep import の増加
- 責務を壊してまで 500 行未満にすること

## 5. 実施手順

1. 500 行超ファイルを一覧化する
2. 各ファイルを責務ブロック単位で切り分ける
3. 先に `types/constants/logic` を抽出する
4. 次に `hook/container` を抽出する
5. 最後に presenter を薄くする
6. `index.ts` 公開 API を整理する
7. `pnpm build` を実行する
8. 再計測し、500 行超が残る場合は残置理由と次の分割候補を記録する

## 6. 完了定義

- 責務境界が明確で、変更容易性が向上している
- ビルドが成功
- 既存テストが必要範囲で成功
- import 解決エラーがない
- 500 行超が残る場合は、無理な分割を避けた理由が説明可能

## 7. テンプレート

```text
feature/
  feature.tsx
  feature.container.tsx
  feature.hook.ts
  feature.logic.ts
  feature.data-access.ts
  feature.constants.ts
  feature.types.ts
  index.ts
```

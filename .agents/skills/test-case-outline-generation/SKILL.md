# Test Case Outline Generation Skill

このスキルは、実装前にテスト観点を固定するための「テストケース設計専用」スキルです。

## 1. いつ使うか

- TDDのREDフェーズに入る直前
- 新機能の受け入れ条件をテスト観点へ分解したい時
- 仕様レビュー用に、生きたドキュメントとしてテスト名を先に作りたい時
- React Hookの状態遷移や副作用を実装前に整理したい時

## 2. 出力ルール（厳守）

1. Vitestの `describe` と `it.todo` のみを使用する
2. 実行ロジック（Arrange/Act/Assert）を一切書かない
3. モックの実装・具体化を書かない
4. 各 `it.todo` の直前に、意図説明コメントを付ける
5. テスト名は日本語で「〜の時、〜であること」に統一する
6. 階層は「機能 → 条件カテゴリ → ケース」の順で構成する

## 3. カバレッジ観点テンプレート

最低限、以下のカテゴリを含める。

- 正常系（Happy Path）
- 異常系（入力不正、範囲外、依存データ不足）
- 境界値（最小・最大・空・同率・0件）
- 仕様制約（件数上限、期間デフォルト、画面配置）
- パラメータ変更時（期間指定、並び順、フィルタ）
- 時刻・日付境界（日跨ぎ、タイムゾーン、開始日/終了日）

## 4. React Hook向け追加観点

Hookを対象にする場合、以下を追加する。

- 初期状態（初回レンダー時）
- 状態遷移（入力更新、再計算、リセット）
- 依存値変更時の再評価
- 非同期完了前後の状態
- エラー状態と復帰
- アンマウント時の後始末
- Strict Modeでの再実行耐性（副作用の重複実行を想定）

## 5. 生成フォーマット例

```ts
import { describe, it } from "vitest";

describe("[対象機能名]", () => {
  describe("正常系", () => {
    // 基本入力で期待される結果が返ることを確認する
    it.todo("通常入力の時、期待した結果であること");
  });

  describe("境界値", () => {
    // 上限件数や最小値の境界で壊れないことを確認する
    it.todo("上限件数に達した時、仕様どおりに切り詰められること");
  });
});
```

## 6. 不明点がある時の扱い

以下が曖昧な場合は、ケース生成前に必ず確認する。

- 期間の解釈（タイムゾーン、inclusive/exclusive）
- 同率順位の扱い
- データが0件の時の表示仕様
- フロント指定可能なパラメータ範囲
- APIエラー時のUI仕様

## 7. 本依頼（テンプレート利用）への適用メモ

「テンプレート利用ランキング（上位20件、デフォルト過去7日、フロント期間指定、`/dashboard`表示）」では、最低でも次を含める。

- 上位20件への切り詰め
- 7日デフォルト範囲の集計
- 任意期間指定時の再集計
- 同率順位の表示ルール
- 範囲内データ0件時の表示
- `/dashboard` での表示要件

## 8. 運用フック（任意）

テストケース作成の依頼文に以下を先頭付与して、このスキルを呼び出す。

- 「test-case-outline-generation スキルを適用し、describe + it.todo のみで出力してください。」

これにより、TDD開始時に毎回同じ品質でケース設計を再現できる。

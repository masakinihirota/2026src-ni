# Barrel Import Boundary Guard - Agent Instructions

## 目的

**各ページが本当に独立した小さなミニアプリとして機能しているか** を確認すること。

複数ページが無意識のうちに共有コンポーネント・ロジックに依存していないか、バレルインポート構造が正しく機能しているかを監査します。

## このスキルを使う場面

- ページ間の結合度が高くなっていないか確認したい
- アーキテクチャの独立性が壊れる前に検知したい
- 新機能追加時にページ構造が守られているか検証したい
- リリース前に「本当にミニアプリ化できているか」を整理したい

## 実行プロトコル

1. `pnpm ci:barrel-isolation:report` を実行
2. 全 125+ エントリーポイントをスキャン
3. 各ページの `@/components/*` インポートを分析：
   - **Compliant** → ページ専用 barrel のみ依存（独立している）
   - **Warning** → 深い階層に直接依存（改善の余地）
   - **Violation** → 内部実装に直接依存（独立していない）
4. 結果から「ページ間の独立性スコア」を計算
5. 修正が必要な場合は優先度付けして提案

## 必須ルール

1. **独立性を第一に考える**: インポート分類は「ページが独立しているか」の観点から判定
2. **Violation は即改善対象**: lib/hooks/_internal への依存は複数ページから使われると共有ロジックになり、独立性を完全に失う
3. **Warning は段階導入**: 深い階層のコンポーネント直接依存は、段階的に sub-barrel 化して改善
4. **スコア化する**: Compliant 件数から独立性の度合いを数値化し、進捗を可視化

## 独立していないパターン（Violation）

- `@/components/<group>/lib/**` → 複数ページから使われると共有ロジックに
- `@/components/<group>/hooks/**` → 複数ページから使われるカスタムフック
- `@/components/<group>/_internal/**` → 複数ページから使われる内部実装

## 独立している状態（Compliant）

- `@/components/<group>` → グループ全体の公開 API
- `@/components/<group>/<page-unit>` → ページ専用の sub-barrel

## 最低限の成果物

- **独立性スコア**: Compliant / Warning / Violation の件数と割合
- **独立性判定**: 100% Compliant に向けた改善ステップ
- **修正優先度リスト**: 特に Violation の緊急度を明示
- **段階導入ロードマップ**: Phase 1/2/3 でどの Warning を改善するか提案
- **具体的な修正提案**: sub-barrel 化の検討など、改善方向を示唆

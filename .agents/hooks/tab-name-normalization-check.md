# Tab Name Normalization Check Hook

タブ名を扱う処理（追加・編集・リネーム）を変更した場合は、以下を必ず実行してください。

## トリガー条件

- `tabName` の初期値・表示名を変更したとき
- `addTab` / `renameTab` / `handleConfirmEditTabName` 相当を変更したとき
- タブ追加時に名称を自動生成している処理を変更したとき
- 「同名タブができる」「意図しない連番が付く」不具合を修正するとき

## 1. 正規化ルールの確認

1. 先頭/末尾空白は `trim()` で除去する。
2. 空文字は採用しない（元の名前を維持、または追加を拒否）。
3. 同一判定の基準（完全一致・大文字小文字・全半角など）を実装コメントで明示する。

## 2. 重複禁止ルールの確認

1. 追加時: 既存名と重複する場合は追加しない。
2. 編集時: 既存名と重複する場合は変更しない。
3. 追加/編集の双方で、同じ判定ロジックを再利用する。

## 3. 共通化ルールの確認

1. `tabName` の正規化・重複判定はロジック層へ寄せる。
2. 画面コンテナ側は「結果を反映するだけ」に留める。
3. 同種画面へ横展開できるよう、ユーティリティ化候補を残す（関数分離 or 共通モジュール）。

## 4. テスト最小要件

- 初期タブ名の期待値テスト（例: `自分の作品`）
- 追加時の重複拒否テスト
- 編集時の重複拒否テスト
- 空文字編集の維持テスト

## 5. 変更後の最小検証

PowerShell 例:

```powershell
pnpm test:file -- src/components/work/work-profile-tabs/work-profile-tabs.logic.test.ts src/components/work/work-profile-tabs/work-profile-tabs.container.test.tsx
pnpm build
```

## 6. 失敗時の対処

- UIでしか重複判定していない場合は、必ずロジック層へ移す。
- 追加と編集で判定基準がズレている場合は、同一関数へ統合する。
- 既存機能と要件が衝突する場合は、どちらを正本にするかをルールへ追記する。
